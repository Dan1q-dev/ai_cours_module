# Отчёт о состоянии AI-модуля

Дата: 2026-03-17

## 1. Цель работы

В рамках текущего этапа был собран и доведён до рабочего состояния AI-модуль для образовательной платформы. Основная задача состояла в том, чтобы реализовать API-first контур для:

- RAG-ответов по учебным материалам
- потоковой генерации ответов
- индексирования учебных материалов
- контроля безопасности, лимитов и телеметрии

Тестовая веб-страница используется как стенд для локальной проверки, однако основной результат работы представляет собой именно API-модуль, который может быть интегрирован в основной backend.

## 2. Что реализовано

### 2.1. RAG pipeline

Реализован полный базовый цикл:

- `retrieval` — поиск релевантных фрагментов по embeddings и keyword scoring
- `prompt building` — формирование запроса к LLM с изоляцией системных инструкций и контекста
- `generation` — генерация ответа моделью только на основе найденного контекста

Особенности текущей реализации:

- используется hybrid scoring
- есть controlled no-answer сценарий, если в материалах нет ответа
- retrieved context помечается как `UNTRUSTED CONTEXT`

Статус: **готово**

### 2.2. Стриминг ответов

Реализован потоковый ответ через `SSE` в `/api/chat`.

Поддерживаются события:

- `meta`
- `chunk`
- `done`
- `error`

Это позволяет получать ответ по частям и строить realtime UX без ожидания полного завершения генерации.

Статус: **готово**

### 2.3. Источники и цитирование

Реализовано возвращение источников ответа:

- `sources`
- `citations`
- `locateSnippet`

Источники строятся на базе реально извлечённых retrieval chunks, а не постфактум по тексту модели.

Статус: **готово**

### 2.4. Guardrails, модерация и prompt-injection защита

Реализован локальный слой защит:

- пустой ввод
- слишком длинный ввод
- повреждённый/повторяющийся ввод
- очевидные prompt-injection паттерны
- базовые unsafe-категории

При срабатывании блокировки модуль возвращает структурированный ответ без обращения к генерации.

Статус: **готово как integration v1**

Ограничение:

- текущая модерация rule-based
- classifier-grade safety слой пока не подключён

### 2.5. Квоты, лимиты, cost control

Реализованы:

- лимит запросов в минуту
- дневной лимит запросов
- дневной лимит токенов
- месячный лимит по оценочной стоимости

Также ведётся учёт usage и estimated cost.

Статус: **готово**

Ограничение:

- без Redis это single-instance реализация
- для распределённого production-кластера потребуется усиление

### 2.6. История диалогов и телеметрия

Реализовано хранение в `PostgreSQL` через `Prisma`:

- `sessions`
- `messages`
- `request_events`
- `quota_counters`

Есть API для:

- истории диалога
- usage summary
- telemetry / diagnostics

Также реализован redaction чувствительных данных.

Статус: **готово**

### 2.7. Обработка учебных материалов

Поддержана загрузка и обработка:

- `PDF`
- `PPTX`
- `DOCX`
- `TXT / MD / CSV`
- `video/audio` через локальный `STT`

Реализовано:

- извлечение текста
- извлечение базовой структуры
- chunking по заголовкам и абзацным группам
- генерация embeddings
- индексирование материалов
- versioned course indexes
- активация версии индекса

Статус: **готово как working ingestion v1**

Ограничения:

- нет OCR для scan-PDF
- нет layout-aware/table-aware ingestion enterprise-уровня
- ingestion пока синхронный, без async job queue

## 3. Что готово полностью

На текущем этапе можно считать завершёнными:

- RAG (retrieval -> prompt -> generation)
- SSE streaming
- sources/citations
- history
- usage
- telemetry
- prompt-injection базового уровня
- quotas/limits/cost control
- indexing и version activation для учебных материалов

## 4. Что ещё не доведено до полного enterprise-final

Следующие пункты остаются открытыми:

1. Вынесение embeddings/index retrieval в полноценный production vector store
   - сейчас используется рабочий локальный индексный слой
   - желательно перейти на `pgvector` или отдельный vector DB

2. Усиление ingestion pipeline
   - OCR
   - более точный parsing таблиц, схем, сложной верстки
   - async jobs с отдельным статусом

3. Усиление safety layer
   - classifier-based moderation
   - более сильная prompt-injection защита

4. Redis runtime layer
   - в коде подготовлен optional слой
   - фактически Redis пока не подключён

5. Production observability
   - сейчас есть локальная telemetry
   - нет внешней аналитической/мониторинговой инфраструктуры

## 5. Фактический статус готовности

На текущий момент AI-модуль находится в состоянии:

**Integration-ready v1**

Это означает:

- модуль можно подключать к основному backend
- API-контракт уже существует
- основные сценарии работают
- тестовый прогон проходит успешно

Но это ещё не означает:

- что ingestion доведён до enterprise-grade обработки любых документов
- что quota/runtime слой готов к распределённой нагрузке
- что safety/middleware часть исчерпывающая для крупного production без следующего этапа усиления
